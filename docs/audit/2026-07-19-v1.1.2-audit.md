# Vani v1.1.2 — Security & Toolchain Audit

**Date**: 2026-07-19
**Version audited**: 1.1.2 (patch — cyrius pin `6.4.49` → `6.4.67`, zero source change)
**Kernel baseline**: `linux-api-headers 7.1-1`, running kernel `7.1.3-arch1-3`
**Prior audit**: [`2026-07-06-v1.0.0-audit.md`](2026-07-06-v1.0.0-audit.md)
**Scope**: toolchain + stdlib dependency delta, ALSA UAPI ABI re-pin, external
CVE sweep. Method: six parallel analysis lenses, each finding adversarially
re-derived by an independent verifier against live source; the load-bearing
results were then re-measured by hand (see *Reproduction* below).

---

## Verdict

**Clean.** No source change is justified by this sweep. 1.1.2 ships as a pure
toolchain/dependency patch.

The strongest structural result: **cycc version has no effect on vani's emitted
bytes.** A 2×2 A/B varying `(cycc version) × (stdlib snapshot)` independently
(hybrid `CYRIUS_HOME` prefixes, pin aliased to a common label) yields

| cycc | stdlib | `vani_smoke` |
|---|---|---|
| 6.4.49 | 6.4.49 | 456,072 B — md5 `06b46209…` |
| 6.4.67 | 6.4.49 | 456,072 B — md5 `06b46209…` |
| 6.4.49 | 6.4.67 | 468,456 B — md5 `e1932270…` |
| 6.4.67 | 6.4.67 | 468,456 B — md5 `e1932270…` |

Every codegen, linker, and DCE change in 6.4.50–6.4.67 is therefore *provably*
inert on vani's source; the entire +12,384 B delta is stdlib growth.

> **Methodology note.** `cyrius build` resolves stdlib from
> `~/.cyrius/versions/$PIN/lib`, **not** the repo's gitignored `lib/`. Verified
> by poisoning `lib/alloc.cyr` with a canary function: the build was byte-identical
> and the canary never appeared. A naive A/B that only swaps the manifest pin
> therefore moves *both* axes at once and cannot isolate the compiler — the
> matrix above exists specifically to avoid that trap.

## 1. ALSA UAPI ABI re-pin

Vani hardcodes ioctl numbers, struct sizes, and field offsets — kernel UAPI is
the spec, so drift is a correctness *and* memory-safety concern. Re-derived from
scratch with a generated C probe against the installed kernel headers.

**All 18 pinned ioctl numbers match — 0 mismatches:**

`HW_REFINE 0xc2604110`, `HW_PARAMS 0xc2604111`, `SW_PARAMS 0xc0884113`,
`PREPARE 0x4140`, `RESET 0x4141`, `START 0x4142`, `DROP 0x4143`,
`DRAIN 0x4144`, `PAUSE 0x40044145`, `RESUME 0x4147`, `WRITEI 0x40184150`,
`READI 0x80184151`, `STATUS 0x80984120`, `CARD_INFO 0x81785501`,
`ELEM_LIST 0xc0505510`, `ELEM_INFO 0xc1105511`, `ELEM_READ 0xc4c85512`,
`ELEM_WRITE 0xc4c85513`.

**All 8 pinned struct sizes match the kernel:** `snd_pcm_hw_params` 608,
`snd_pcm_sw_params` 136, `snd_pcm_status` 152, `snd_xferi` 24,
`snd_ctl_card_info` 376, `snd_ctl_elem_list` 80, `snd_ctl_elem_info` 272,
`snd_ctl_elem_value` 1224.

Mapping those onto in-tree **buffers**, six match exactly — `hwp[608]`
(`src/alsa.cyr:579,725`, `src/device.cyr:347`), `swp[136]`
(`src/alsa.cyr:651`), `xferi[24]` (`src/alsa.cyr:831,883` — the size the
2026-04-30 audit's `[16]` got wrong), `info[272]` (`src/mixer.cyr:145,182,221`),
`list[80]` (`src/mixer.cyr:310`), `val[1224]`
(`src/mixer.cyr:163,198,235,254`) — and two do not:

- **`snd_pcm_status` (152) is buffered as `var status[192]`** at
  `src/alsa.cyr:920`, and the comment at `:910` asserts "snd_pcm_status is 192
  bytes" — contradicting both the constant table at `:79` ("152 bytes → 0x098")
  and the kernel probe. **Over-allocated, so it is safe** (the kernel writes
  152 into a 192-byte zeroed buffer and `audio_get_state` reads offset 0), but
  the inline comment is wrong and is exactly the stale-comment rot this sweep
  exists to catch. Filed as finding #3 below.
- **`snd_ctl_card_info` (376) has no buffer** — `SNDRV_CTL_IOCTL_CARD_INFO` is
  defined (`src/mixer.cyr:33`) and its encoding *is* test-pinned
  (`tests/tcyr/vani.tcyr:443`, `:453`), but the ioctl is never issued.

The v1.0.0 `PAUSE` correction survives. No UAPI drift since the last audit: the
running kernel moved 7.0.14 → 7.1.3 but the header package did not.

## 2. External CVE sweep

Searched NVD for CVEs published **2026-06-25 → 2026-07-19** under ALSA / snd /
PCM / usb-audio / hda / cyrius. **Eight** in-window Linux kernel audio CVEs
(plus three Windows `usbaudio.sys` entries, listed for completeness but N/A —
vani has no Windows target); **none triggerable through vani's ioctl surface.**

| CVE | Area | Disposition |
|---|---|---|
| CVE-2026-64134 | `sound/core/pcm_lib.c` — bogus `iov_iter` for silencing | **Closest approach** — the only in-window bug in the generic PCM transfer path vani drives. Doubly unreachable: the kernel gates on `runtime->silence_size > 0`, and vani hard-zeroes `SWP_SILENCE_THRESHOLD` + `SWP_SILENCE_SIZE` in every SW_PARAMS (`src/alsa.cyr:658-659`, intentional per the contract comment at `:636`); and the NULL deref is RISC-V-specific while vani ships x86_64/aarch64. |
| CVE-2026-64001 | `sound/core/oss/pcm_oss.c` — setup-list UAF | OSS compat layer; needs `/dev/dsp*`. vani opens only `/dev/snd/pcmC{n}D{m}{p,c}` and never reads procfs. |
| CVE-2026-64029 | `seq_ump_client.c` — UMP teardown UAF | seq/rawmidi. vani issues only the `SNDRV_PCM_*` and `SNDRV_CTL_*` sets; no SEQ/RAWMIDI numbers exist in-tree. |
| CVE-2026-64133 | `sound/pci/asihpi` — OOB array read in control cache | Reached beneath `ELEM_READ`, which vani *does* issue — but `_ctl_elem_id_init` memsets the struct and sets only iface + name, leaving `numid`/`index` at 0 (`src/mixer.cyr:124-126`), so vani cannot supply an out-of-range index. No vani target is asihpi hardware. |
| CVE-2026-53350 / -53348 / -64107 | ASoC wm_adsp, SDCA, pcm512x | Driver bind/unbind/probe paths. vani performs no enumeration, probe, bind, or card unregister — yukti owns discovery. |
| CVE-2026-53291 | hda/conexant — missing `IS_ERR` in `cx_probe` | Published 2026-06-26, not in the v1.0.0 table; picked up here. Probe path, N/A. |
| CVE-2026-49794 / -50453 / -58528 | Windows `usbaudio.sys` OOB read | Zero new Linux `snd-usb-audio` CVEs in window. vani has no Windows target. |

**Toolchain**: cyrius has zero NVD entries. The 6.4.49 → 6.4.67 stdlib delta
contains exactly one memory-corruption-class item set (the patra/thread_local
slot rework) and it compiles into vani's closure as a yukti transitive that is
never executed.

## 3. Dependency delta

- **yukti 2.2.9 → 2.2.10** — resolved bundle diffs by exactly one line (the
  version stamp). `vani_open_yukti`'s call surface is bit-stable.
- **patra 1.12.9 → 1.12.12** — genuinely additive; CAS-gated migration off
  hardcoded thread-local slots onto 6.4.65's `thread_local_alloc()` (based at
  16, above the frozen 0-15 app range). Reaches vani only via yukti's
  `device_db`, which vani never enters; DCE-strips entirely. **vani hardcodes
  no thread-local slot**, so the collision class the allocator closes never
  applied here.
- **18 of the 40 resolved modules differ** between the two trees. Besides yukti
  and patra: `thread_local` (the module carrying the 6.4.65 allocator work),
  `alloc` + `alloc_agnos`/`_macos`/`_windows`, `chrono`, `io`, `sakshi`,
  `syscalls` + all arch/OS variants, `args_macos`. Still 40 modules — no new
  stdlib dependency; `[deps] stdlib` unchanged.
- **`ALLOC_MAX` 256 MiB → 2 GiB** (landed in **6.4.50** — the installed 6.4.50
  snapshot already carries `0x80000000`, though the stdlib's own inline comment
  credits v6.4.51) is the single runtime behavior delta
  in the whole bump — see CHANGELOG. Affects `vani_ring_new` only in
  (256 MiB, 1 GiB]; fails in the safe direction.

## 4. Findings carried forward (pre-existing, not caused by this bump)

Each was confirmed byte-identical before and after the bump. **None is a 1.1.2
regression**; all are filed for the next minor.

| # | Severity | Location | Finding |
|---|---|---|---|
| 1 | LOW | `src/buffer.cyr:47-48` | `vani_ring_new` stores both the 40-byte header and the payload from `alloc()` without a NULL guard, then dereferences `rb` unchecked. Bites only on true OOM. Fix: `if (rb == 0) { return 0; }` + a checked local for the payload. |
| 2 | LOW (defense-in-depth) | `src/mixer.cyr:167-169`, `:239-241` | `vani_mixer_set_volume` / `_set_mute` use the kernel-supplied ELEM_INFO `count` directly as a `store64` bound into the 1224-byte local `val[1224]` from base 72; first OOB store at `count ≥ 145`. Volume guards only `count == 0` (`:158`); mute guards nothing. **Not a vulnerability** — only the kernel driver sets `count`, and it already outranks the caller; the one attacker-adjacent vector (USB-audio) is capped at `MAX_CHANNELS = 16` by `sound/usb/mixer.c`. Linux-only (agnos fails closed at `:86-91`). Same class heap-side in `vani_mixer_list_elements` (`:317`, `:340` — trusts `count` then `used` with no cross-check against the allocated cap). Fix: clamp `count > 128` → `VANI_ERR_MIXER_ELEM`, add a `used <= count` check, pin with a tcyr assertion on `72 + 128*8 <= 1224`. |
| 3 | INFO | `src/alsa.cyr:910` vs `:79` | **Stale comment contradicts the pinned table.** `audio_get_state` declares `var status[192]` (`:920`) under a comment claiming "snd_pcm_status is 192 bytes", but the constant table at `:79` says 152 and the kernel probe confirms 152. Over-allocated, therefore safe — but it is a wrong comment sitting on a UAPI-pinned buffer, the same shape as the PAUSE encoding that rotted unnoticed until v1.0.0. Fix the comment (or shrink the buffer to 152 and pin it with a tcyr assertion, matching how `hwp`/`swp`/`xferi`/`val` are handled). |
| 4 | INFO | `dist/vani.deps` | Omits `freelist`, `process`, `patra`. A consumer resolving via `[deps.vani] modules = ["dist/vani.cyr"]` gets 13 cosmetic `undefined function` warnings out of `lib/yukti.cyr`; builds succeed, all sites DCE-dead. The bundled-stdlib path is unaffected. |
| 5 | INFO | `lib/yukti.cyr` (upstream) | The agnos build emits **15** warnings, all from stdlib yukti — 8 duplicate-symbol (`SYS_SOCKET`/`CONNECT`/`BIND`/`RECVFROM`/`SETSOCKOPT`/`PPOLL`/`STATFS`/`NEWFSTATAT` — reported at compiler lines `lib/yukti.cyr:49-57`, which are post-preprocessor positions; the enum members sit at `:63-71` in the file as committed), 6 syscall-arity (`sys_mount` ×3, `sys_rmdir`, `sys_unlink`, `sys_stat`), 1 undefined `sys_umount2`. All in storage/network enumerators vani never calls; all dead-strip. **Identical set under yukti 2.2.9 and 2.2.10**, A/B'd via hybrid `CYRIUS_HOME` prefixes. Upstream yukti item. Corrects 1.1.1's unqualified "agnos builds clean". |
| 6 | INFO | `.github/workflows/ci.yml:101` | The distlib drift gate covers the two `.cyr` bundles but not the `.deps` sidecars. Effectively self-healing — every release bumps the `# Version:` stamp inside both `.cyr` files, forcing a `cyrius distlib` run that rewrites the sidecars in the same operation. Low-value hardening. |
| 7 | INFO | `src/alsa.cyr:806` | cyrlint reports 1 untracked deferral (an audit follow-up comment lacking a CHANGELOG/issue cross-reference). Present identically under both pins; the deferral counter is independent of the warning counter, so vani's "0 warnings" claim is not falsified. Optional `--strict-deferrals` adoption next minor. |

## 5. Reproduction

```bash
# ABI re-pin (probe source in the audit working set)
gcc -o abi abi.c && ./abi        # 18 ioctls + 8 struct sizes vs src/*.cyr

# 2x2 compiler-inertness matrix — vary cycc and stdlib independently
for CC in 6.4.49 6.4.67; do for LL in 6.4.49 6.4.67; do
  P=prefixes/cc${CC}_lib${LL}; mkdir -p $P/versions/9.9.9
  cp -r ~/.cyrius/versions/$CC/bin $P/versions/9.9.9/bin
  cp -r ~/.cyrius/versions/$LL/lib $P/versions/9.9.9/lib
done; done
# with cyrius.cyml pinned to the common label 9.9.9:
CYRIUS_HOME=$P CYRIUS_DCE=1 $P/versions/9.9.9/bin/cyrius build programs/smoke.cyr out

# stdlib resolution proof — local lib/ is NOT used
echo 'fn __poison_canary_zz(): i64 { return 1; }' >> lib/alloc.cyr
CYRIUS_DCE=1 cyrius build programs/smoke.cyr out   # byte-identical; canary absent
```

## 6. P(-1) disposition

1.1.2 is a **patch**, so the full P(-1) sweep (which the roadmap triggers before
every *minor* bump) is not owed. Discharged anyway: cleanliness gates, test
sweep, distlib drift, bench baseline, external CVE research, and this document.
Not done: deep internal review pass, new regression assertions, downstream
consumer rebuilds. Recorded here so the skip is a decision, not an omission.
