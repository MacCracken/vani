## Internal deep review

### `[lib.core]` profile correctness

The new profile lists exactly one module:
`modules = ["src/alsa.cyr"]`. Pre-cut verification confirmed
`src/alsa.cyr` is fully self-contained — zero cross-module
references. This is checked manually (grep against the file's
identifier set) and verified by the `cyrius distlib core` run
not emitting an "unresolved" error against any non-stdlib
symbol.

The bundle's standard "unresolved symbols (expected; consumer
supplies stdlib)" note matches the full bundle. Stdlib pulls
required by `src/alsa.cyr` for the core profile:
- `syscalls` — `sys_open`, `sys_close`, `sys_write`,
  `syscall(SYS_IOCTL, ...)` (all migrated to stdlib wrappers
  in 0.9.0).
- `alloc` — `alloc()` for the `audio_h` handle struct.
- `string` — `strlen` for path-builder boundary.

Consumers using the core profile must list at minimum these
stdlib modules in their own `[deps] stdlib`. The same set
suffices for any consumer that previously used the full bundle
without the `vani_*` higher-level API; for cyrius-doom, the
already-listed stdlib (15 modules) is a strict superset.
