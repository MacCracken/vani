## Findings

### HIGH / CRITICAL — none

### LOW-1 (fixed this cut) — `SNDRV_PCM_IOCTL_PAUSE` mis-encoded (dormant)

- **File**: `src/alsa.cyr:90` (enum `AlsaIoctl`).
- **Class**: ioctl `_IOC` size-field mis-encoding — the same class as the
  2026-04-30 WRITEI/READI/`xferi[24]` HIGH findings.
- **Detail**: PAUSE was hardcoded `0x00404145` (dir `NONE`, size `0x40`=64,
  type `'A'`, nr `0x45`). The kernel defines it `_IOW('A', 0x45, int)` =
  `0x40044145` (dir `WRITE`, size 4). The `64` sat in the size field where
  the WRITE-direction bit + 4-byte size belong.
- **Exploitability**: **dormant / harmless today.** PAUSE is defined in the
  enum but **never invoked** — there is no `audio_pause` / `vani_pause`
  wrapper and no `syscall(..., SNDRV_PCM_IOCTL_PAUSE, ...)` anywhere in
  `src/`, `programs/`, or `tests/`. `SND_PCM_STATE_PAUSED = 6` is a separate,
  correct state constant. A future `audio_pause()` built on the old value
  would have sent a malformed cmd → kernel `-ENOTTY` (the exact WRITEI/READI
  bug the earlier audit fixed).
- **Fix**: corrected to `0x40044145`; pinned by a new regression assertion
  `test_pause_ioctl_encoding` (`tests/tcyr/vani.tcyr`) mirroring the RESUME
  pin. **Not an API change** — `api-surface` counts public functions, not
  enum constants; the surface stays 106.
- **Surfaced by**: the CVE/ABI research sweep for this cut (independent
  recomputation of every hardcoded ioctl number from the current UAPI).

### INFO — `snd_pcm_status` buffer oversized vs current sizeof (no action)

- **File**: `src/alsa.cyr` `audio_get_state` (`var status[192]`).
- Current x86_64 `sizeof(struct snd_pcm_status)` ≈ 152 bytes; vani's buffer
  is 192. The kernel writes ≤ 152 into a 192-byte buffer → **memory-safe**
  (over-read impossible; the `state` field at offset 0 is read correctly).
  The 192 is deliberate defense-in-depth headroom over the UAPI `reserved[]`
  tail. No functional impact; recorded for the paper trail.

## Kernel / CVE sweep (2026-05-01 → 2026-07-06)

vani's issued-ioctl set: on PCM fds — HW_PARAMS, HW_REFINE, SW_PARAMS,
PREPARE, START, DROP, DRAIN, RESUME, WRITEI_FRAMES, READI_FRAMES, STATUS;
on control fds — CARD_INFO, ELEM_LIST, ELEM_INFO, ELEM_READ, ELEM_WRITE.
vani does **not**: scan `/dev/snd` (yukti owns discovery), parse USB
descriptors, link streams (LINK/UNLINK), create userspace controls
(ELEM_ADD/REMOVE), use mmap access, or touch the OSS/aloop paths.

| CVE | Component | Severity | Verdict for vani |
|-----|-----------|----------|------------------|
| CVE-2026-53242 | `sound/core/pcm_native.c` `snd_pcm_drain()` wait-queue corruption | HIGH 7.8 | **Reached-but-not-triggerable.** vani issues DRAIN (`audio_drain`), but the corruption occurs only on **linked** streams racing an UNLINK. vani is single-stream and never issues LINK/UNLINK → the drain group is always a singleton; the precondition is unreachable. Watch if stream-linking is ever added. |
| CVE-2026-46088 | `sound/core/control.c` `snd_ctl_elem_init_enum_names()` via ELEM_ADD | MED 5.5 | **N/A.** Reached only via `SNDRV_CTL_IOCTL_ELEM_ADD` (userspace-created controls); vani issues only ELEM_LIST/INFO/READ/WRITE, never ELEM_ADD. |
| CVE-2026-43279 | `sound/usb/endpoint.c` `prepare_silent_urb()` OOB write | HIGH 7.8 | **N/A.** Kernel-internal snd-usb-audio URB path; vani never touches URBs — it hands the driver PCM ioctls and the driver marshals URBs. |

**PulseAudio / PipeWire CVEs**: N/A by construction — vani is ioctl-only,
no daemon dependency. Recorded rather than skipped.

## Toolchain delta screening (cyrius 6.4.4 → 6.4.10)

The entire `6.4.x` delta vani adopts is the **SIMD-compute arc** (f32v4,
f32 matmul, integer SIMD, f32v8/AVX2) — every entry is `api-surface +N /
−0` (purely additive) and vani uses **no SIMD**, so nothing in the arc
reaches vani code. Two non-SIMD items screened:

- **6.4.10 P1 — bare top-level `var X[N]` 8× under-size fix**: a memory-
  safety fix, but a **no-op for vani** — vani has **zero** top-level bare
  arrays. All 16 `var X[N]` in `src/` are function-local ioctl-struct byte
  buffers (272 / 608 / 136 / 192 / 24 / 80), which keep N-byte sizing under
  both old and new cycc. Confirmed by grep.
- **6.4.10 distlib per-module read cap 256 KB → 1 MB**: no impact — vani's
  bundles are 80.6 KB / 33.1 KB, far under even the old cap.
- **6.4.4 f32v4 field-OOB / param-confusion fixes**: SIMD-only; vani unaffected.

No security-tagged cyrius entry in the delta touches vani.

## Input-validation / correctness spot-review

- **XRUN recovery** (`playback.cyr` `vani_play`, `capture.cyr` `vani_record`):
  bounded **single** re-prepare-and-retry, state-classified (XRUN vs
  SUSPENDED vs propagate); no unbounded loop, no `break`-in-`while`-with-`var`.
- **Ring buffer** (`buffer.cyr`): pow-of-2 `& mask` wrap; `_next_pow2` capped
  at 1 GiB (`VANI_RING_MAX_BYTES`, prior LOW-1 fix); `vani_ring_new` guards
  `capacity <= 0` and `> MAX`.
- **Format math** (`format.cyr`): `vani_bytes_to_frames` guards `bpf == 0`;
  `frames * bytes_per_frame` bounded by `AUDIO_FRAMES_MAX` (2^28) × ≤32 bpf,
  well within i64.
- **Untrusted external data** (`device.cyr` `vani_open_yukti`): validates
  `desc != 0`, dispatches only on PLAYBACK/CAPTURE, rejects out-of-range
  direction; card/dev are integer-formatted into the device path (no
  traversal / injection surface).

## Verdict

Swept clean as of 2026-07-06 against kernel 7.0.14 and the ALSA UAPI
`SNDRV_PCM/CTL_VERSION` current at survey. One dormant LOW finding (PAUSE
ioctl number) fixed and regression-pinned; one INFO note (status buffer
headroom). No HIGH+ findings. No new ALSA / sound CVE since the prior
sweep is triggerable through vani's ioctl surface. The full `vani_*`
