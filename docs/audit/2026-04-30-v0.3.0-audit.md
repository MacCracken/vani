## Findings

### LOW-1 — CI lacked defense-in-depth lock-file presence guard (FIXED in this sweep)

**Class**: supply-chain hardening.
**File**: `.github/workflows/ci.yml`.
**Symptom**: CI ran `cyrius deps` then `cyrius deps --verify`
without first asserting `cyrius.lock` was present. If an
accidental `.gitignore` addition or commit revert dropped the
lock from the repo, fresh CI runs would silently regenerate it
from whatever the [deps.yukti] tag (`2.2.1`) currently points at,
then verify against the just-created lock — providing zero
integrity. Vani's lock IS committed today (commit `5d331f5`), so
this is defense-in-depth, not an active issue.
**Real-world reach**: theoretical. Mitigated as long as the
human-side review catches accidental lock removal. Since the
fix is one shell line, no reason to leave it implicit.
**Severity**: LOW — defense-in-depth.
**Status**: fix landed in this sweep. New "Lock file present"
step inserted before `cyrius deps` runs, with explicit
`::error::` and exit 1 on absence.

## External CVE / 0-day research (web)

Prior sweep (2026-04-30, earlier today) surveyed CVE database
through the same date. No realistic new ALSA / sound subsystem
CVE drops in the intervening window.

The deferred items from the prior sweep — YUKTI-DEFER-1 (filter
aloop devices for pro-audio consumers, mitigates CVE-2026-23191)
and YUKTI-DEFER-2 (treat freshly-plugged USB audio as untrusted,
mitigates CVE-2024-53150 / -53197 / -53208 family) — are now
re-evaluated against the v0.3.0 yukti integration:

- **YUKTI-DEFER-1**: yukti 2.2.1 exposes `yukti_audio_driver(desc)`
  which returns the kernel driver name string ("Loopback" for
  snd-aloop, "HDA-Intel", "USB-Audio", etc.). Consumers wanting
  to filter virtual cards walk `yukti_audio_devices()` and
  filter on this field. **Vani's posture**: do not bake the
  policy into the library — consumer decides. This matches the
  user's stated principle for yukti integration ("device is
  yukti's domain"). Documented for downstream consumers.
- **YUKTI-DEFER-2**: the USB descriptor parsing CVEs are
  kernel-side issues. By the time vani's `open()` runs, the
  kernel has already accepted (or rejected) the USB device.
  yukti 2.2.1 added the `audio_devices` device_db table with
  `first_seen` / `last_seen` epochs that consumers can use for
  trust decisions ("never seen this DAC before — prompt the
  user"). Vani has no role in this decision. Documented; no
  vani-side action.

Both items remain consumer-policy concerns rather than vani
fixes. No code change in this sweep.

## Internal deep review

### `vani_open_yukti(desc)` correctness — `src/device.cyr:277`

Edge cases reviewed:
- ✓ `desc == 0` → `VANI_ERR_YUKTI_DESCRIPTOR` early return.
- ✓ `direction == YUKTI_AUDIO_PLAYBACK` (0) → routes to
  `audio_open_playback`.
- ✓ `direction == YUKTI_AUDIO_CAPTURE` (1) → routes to
  `audio_open_capture`.
- ✓ `direction` outside {0,1} → `VANI_ERR_YUKTI_DESCRIPTOR` with
  detail "yukti direction out of range".
- ✓ `card` or `device` negative or > 99 → defense-in-depth chain:
  `_audio_devpath` (prior audit MED-1 fix) caps at 99 with null
  return; `audio_open_*` returns 0; `_vani_device_wrap` returns
  `VANI_ERR_DEVICE_OPEN`. Correct shape for a corrupt descriptor.
- ✓ Non-yukti pointer passed by mistake (random heap alloc):
  `load64` reads at fixed offsets return arbitrary values, which
  funnel through the same defense-in-depth chain. Worst case is
  `VANI_ERR_DEVICE_OPEN` or opening a coincidentally-valid
  card/device. Not a security shape.

**Subdevice handling** (`yukti_audio_subdevice(desc)`): yukti
exposes the field but vani ignores it. At the raw ioctl level
(`open("/dev/snd/pcmC{N}D{M}{p|c}")`), the kernel auto-selects
an available subdevice — there's no UAPI surface to request a
specific one without going through libasound's device-string
parsing. Documented limitation, not a bug. Most cards expose
only subdevice 0; multi-subdevice setups (HDMI multichannel
routing, some pro-audio interfaces) fall through to kernel
auto-select. If a future consumer needs explicit subdevice
routing, it'd be a stdlib audio.cyr addition (not here).

**Verdict**: tight. No findings.

### Supply-chain — `[deps.yukti]` git override

The yukti 2.2.1 git pin (cyrius 5.7.48 still ships yukti 2.1.1
in its bundled `lib/`) brings real supply-chain risk surface:
- Tag could be force-moved upstream.
- Repository compromise.
- First-time consumer with no lock vulnerable to MITM on resolve.

**Mitigations in place**:
- `cyrius.lock` SHA256-pins each resolved file (yukti, sakshi,
  patra). Committed at `5d331f5`.
- `cyrius deps --verify` step in CI rejects any drift from
  locked hashes.
- New "Lock file present" CI step (this sweep, LOW-1 fix)
  asserts the lock exists before `cyrius deps` runs — guards
  against accidental .gitignore / removal.
- The override comment in `cyrius.cyml` documents that this
  is temporary until cyrius re-bundles yukti ≥ 2.2.1, at
  which point the git override gets removed and yukti goes
  back into the `stdlib = [...]` list.

**Verdict**: tight. No outstanding findings.
