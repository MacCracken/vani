## Findings

### HIGH-1 — Missing transitive stdlib deps (FIXED in this sweep)

**Class**: build / runtime correctness.
**File**: `cyrius.cyml`.
**Symptom**: `cyrius build` reported `error: undefined function
'patra_result_get_int' (will crash at runtime)` plus four siblings
(`patra_result_free`, `patra_result_get_str`, plus cascading
`fl_alloc`, `fl_free`, `is_dir`, `dir_list`, `exec_vec`,
`exec_capture`).
**Root cause**: stdlib `yukti.cyr` calls `patra_*` (sqlite-style
result accessors) for its internal device cache; `patra.cyr`
itself depends on `freelist`, `fs`, and `process`. Vani's prior
`[deps].stdlib` listed `yukti` and `sakshi` but none of the
transitives. The build linked but any path reaching yukti's
storage queries would have crashed.
**Exposure**: `vani_open_yukti` is a stub returning an error in
v0.1.0, so no live path reached the missing symbols. Would have
become exploitable the moment v0.3.0 wired the real adapter.
**Fix**: added `patra`, `freelist`, `fs`, `process` to
`[deps].stdlib`. Re-resolved `lib/`. Build / test / distlib all
clean. **No code changes** in `src/`.
**Regression**: covered by gate #1 — any future dep drift
re-triggers the warning.

### MEDIUM-1 — Card number ≥ 10 builds wrong device path

**Class**: input handling / correctness; mild security adjacent.
**Files**: `src/alsa.cyr:31` (`_audio_devpath`),
`src/mixer.cyr:46` (`_vani_ctl_path`).
**Symptom**: Both helpers do `48 + (card % 10)` to encode the
card digit. Card 10 produces `controlC0` / `pcmC0D0p` — the same
path as card 0. Same collision for card 20, 30, etc.
**Real-world reach**: ALSA card numbers above 9 are unusual but
real. Setups with multiple USB audio interfaces, audio-over-IP
adapters, or virtual cards (snd-aloop, snd-dummy) routinely hit
card 10+. The vani consumer thinks it's opening card 12 and
silently gets card 2. On systems where card 0 is privileged
(onboard with ALSA mixer), this is a privilege-confusion adjacent
correctness bug.
**Mitigation**: replace the `% 10` shortcut with a proper
decimal itoa. Same fix applies to both helpers. Bounds: ALSA's
`SNDRV_CARDS` is 32 in current kernels, so 2-digit support
suffices for the foreseeable future. Cap at 99 with an explicit
error for safety.
**Severity**: MEDIUM — silent wrong-device routing, not
exploitable from outside but breaks consumers in non-toy setups.
**Status**: fix landed in this sweep — see "Fixes applied" below.

### LOW-1 — `_next_pow2` has no upper bound

**Class**: integer overflow / DoS-shaped.
**File**: `src/buffer.cyr:30`.
**Symptom**: `while (p < n) { p = p * 2; }` — if caller passes
n > 2^62, p doubles past i64 max, becomes negative, loop
condition flips, infinite-loop into a multiplication that wraps.
**Real-world reach**: only a malicious / buggy caller. v0.1.0
public API doesn't expose `_next_pow2` directly; `vani_ring_new`
takes a capacity that the consumer chose. A 2^62-byte ring is
nonsense (4 EB), so this is theoretical.
**Mitigation**: cap input at a sane max (e.g., 1 GiB =
1073741824) with an explicit return-0 error path; document the
cap in the header comment.
**Severity**: LOW — defense-in-depth, no live exploit shape.
**Status**: fix landed in this sweep.

### LOW-2 — Ring buffer monotonic counters never wrap

**Class**: theoretical UB.
**File**: `src/buffer.cyr` (`vani_ring_write`,
`vani_ring_read`).
**Symptom**: `wpos` and `rpos` are monotonic byte counters; we
mask them with `& mask` for indexing but the assignment
`store64(rb + 24, wpos + n)` can overflow i64 after astronomical
volume. At 192 kHz stereo S16 (768 KB/s), i64 wrap takes ~381 K
years. Practically impossible.
**Mitigation**: not landing — would add a counter-modulo step
on every write/read for zero practical benefit. Documented for
posterity only.
**Severity**: LOW — theoretical only.
**Status**: documented; no code change.

### LOW-3 — `vani_ms_to_frames` unbounded multiply

**Class**: integer overflow.
**File**: `src/format.cyr:90`.
**Symptom**: `(rate * ms) / 1000`. At rate 192000 and ms 2^48,
the product overflows. Practical inputs (ms ≤ a few seconds,
rate ≤ 192 kHz) are 10+ orders of magnitude away from overflow.
**Mitigation**: not landing — overflow guard for theoretical
inputs would clutter a hot path.
**Severity**: LOW — theoretical only.
**Status**: documented; no code change.

## External CVE / 0-day research (web)

Surveyed CVE database, kernel changelogs, and security advisories
for ALSA / sound subsystem CVEs published 2024-04 through
2026-04-30. Mapping each to vani's code paths.

### Hits — kernel-side, NOT vani's bug, but worth tracking

#### CVE-2024-53150 — USB-audio descriptor `bLength` OOB read
Out-of-bounds kernel read when a USB sound device declares a
shorter `bLength` than its descriptor body. CVSS 7.1 (NVD) /
7.8 (Ubuntu).
**Vani path**: kernel parses USB descriptors before vani sees
the device node. Vani's `open("/dev/snd/pcmCxDxp")` arrives
after the kernel has already trusted-or-rejected the USB
device. **Not on vani's API surface.**
**Action**: yukti enumeration (when it lands) should treat
freshly-plugged USB sound cards as untrusted on shared
hardware. Documented as a yukti concern, not a vani fix.

#### CVE-2024-53197 — USB-audio `bNumConfigurations` validation
Sibling of CVE-2024-53150. Same exposure model.
**Action**: same as above. N/A in vani's scope.

#### CVE-2025-40269 — USB-audio PCM transfer narrower-int overflow
Buffer-offset / transfer-size arithmetic done in a narrower
integer type than required, producing wrong values. Information
disclosure + DoS. Lives in `sound/usb/pcm.c` transfer logic.
**Vani path**: vani drives PCM transfer via
`SNDRV_PCM_IOCTL_WRITEI_FRAMES` and `READI_FRAMES`. The frame
count we send is the input the kernel computes from. A
malformed frame count from a vani consumer could plausibly
amplify a bug like this.
**Mitigation (defense-in-depth)**: bound `frames` in
`audio_write` / `audio_read` to a sane max
(`frames * bytes_per_frame < 2^31` is more than enough for any
real-world buffer). Even if the underlying kernel is patched,
the bound costs nothing and protects against
not-yet-disclosed siblings.
**Status**: fix landed in this sweep — see "Fixes applied".

#### CVE-2026-23191 — ALSA aloop racy PCM trigger UAF
Race in `snd-aloop` driver between PCM-trigger callback and
substream open/close. Use-after-free; CVSS 6.9–7.8. Patched
mainline 2026-03-20.
**Vani path**: aloop is a virtual loopback ALSA card. If a vani
consumer opens an aloop device (typical card name `Loopback`)
on an unpatched kernel, frequent start/stop calls from
multiple threads could trigger the kernel UAF.
**Mitigation**: vani is single-threaded today, so the race
cannot be triggered from within one process. Multi-process
coordination is the consumer's problem. Documented as a
yukti-adapter concern: when v0.3.0 lands real device
discovery, give consumers a way to filter out virtual cards
when the use case demands real hardware (pro audio, low
latency, security boundary).
**Status**: documented; no code change in v0.1.0.

#### CVE-2026-23208 — USB-audio OOB write (critical)
Latest in the USB-audio descriptor parsing series. Same
exposure model as CVE-2024-53150 / -53197.
**Action**: N/A in vani's scope.

#### CVE-2026-23301 — ASoC SDCA memory allocation flaw
Specific to ASoC SDCA codec class drivers. Not on vani's
ioctl path.
**Action**: N/A.

#### CVE-2025-37979 — Qualcomm LPASS sc7280 buffer overflow
Hardware-specific (Qualcomm SoC, `sound/soc/qcom/lpass-*`).
**Action**: N/A unless vani targets sc7280 (no roadmap item
for it).

### ALSA UAPI struct drift

`include/uapi/sound/asound.h` carries the ioctl numbers and
struct layouts vani's `src/alsa.cyr` and `src/mixer.cyr` mirror.
Diff-checked the upstream header against the constants we
declare — all stable since Linux 4.x:

| Constant in vani | Upstream field | Status |
|------------------|----------------|--------|
| `SNDRV_PCM_IOCTL_HW_PARAMS = 0xC2604111` | `_IOWR('A', 0x11, struct snd_pcm_hw_params)` | stable |
| `SNDRV_PCM_IOCTL_SW_PARAMS = 0xC0884113` | `_IOWR('A', 0x13, struct snd_pcm_sw_params)` | stable |
| `SNDRV_PCM_IOCTL_PREPARE = 0x00004140` | `_IO('A', 0x40)` | stable |
| `SNDRV_PCM_IOCTL_START = 0x00004142` | `_IO('A', 0x42)` | stable |
| `SNDRV_PCM_IOCTL_DROP = 0x00004143` | `_IO('A', 0x43)` | stable |
| `SNDRV_PCM_IOCTL_DRAIN = 0x00004144` | `_IO('A', 0x44)` | stable |
| `SNDRV_PCM_IOCTL_WRITEI_FRAMES = 0x40104150` | `_IOW('A', 0x50, struct snd_xferi)` | stable |
| `SNDRV_PCM_IOCTL_READI_FRAMES = 0x80104151` | `_IOR('A', 0x51, struct snd_xferi)` | stable |
| `SNDRV_PCM_IOCTL_STATUS = 0x80984120` | `_IOR('A', 0x20, struct snd_pcm_status)` | stable |

`struct snd_pcm_hw_params` size is 608 bytes with 64 bytes
reserved for future extensions — vani v0.2.0 will pack this
correctly. The reserved field gives binary forward-compat
runway; today's struct definition will still work against
future kernels.

`AlsaFormat` constants in `src/alsa.cyr` (`SND_PCM_FORMAT_S8 = 0`,

## Fixes applied in this sweep

1. **HIGH-1**: added `patra`, `freelist`, `fs`, `process` to
   `[deps].stdlib` in `cyrius.cyml`. No source changes.
2. **MEDIUM-1**: replaced `card % 10` decimal encoding in
   `_audio_devpath` (`src/alsa.cyr`) and `_vani_ctl_path`
   (`src/mixer.cyr`) with proper 1-2 digit itoa; cap at 99 with
   an early null-path return (the open() that follows then
   fails cleanly).
3. **LOW-1**: capped `vani_ring_new` capacity at 1 GiB
   (`1073741824`); `_next_pow2` now bails to 0 if input exceeds
   the cap.
4. **DEFENSE-IN-DEPTH (CVE-2025-40269 class)**: bounded `frames`
   in `audio_write` / `audio_read` — `audio.cyr` now refuses
   `frames > 2^28` (256 M frames; ~24 hours of 192 kHz mono).
